This page documents `MITM-DomainFronting.json`, the Xray-core configuration file that implements the local TLS interception and Domain Fronting technique. The file is organized into seven top-level sections: `version`, `log`, `policy`, `dns`, `inbounds`, `outbounds`, and `routing`. Together they create a two-phase pipeline that intercepts your browser's TLS connections, decrypts them locally, and re-forwards them to Google's servers using a spoofed SNI that bypasses censorship filters.

## Version requirement

Xray-core **v26.2.6 or later** is required. The `version.min` field enforces this at startup — older cores will refuse to load the config. Make sure your v2rayN or v2rayNG installation ships with a compatible core binary.

## Inbounds

The configuration defines three inbounds. Each serves a distinct role in the MITM pipeline.

| Tag | Port | Protocol | Purpose |
|---|---|---|---|
| `mixed-in` | `10808` | `mixed` | Main proxy entry point for your browser or system |
| `tls-decrypt-h11` | `11666` | `dokodemo-door` | MITM TLS termination for HTTP/1.1 traffic |
| `tls-decrypt-h211` | `11777` | `dokodemo-door` | MITM TLS termination for HTTP/2 + HTTP/1.1 traffic |

**`mixed-in` (port `10808`)** — Set your browser or system proxy to `127.0.0.1:10808`. This inbound accepts both HTTP and SOCKS5 connections on the same port. Sniffing is enabled with `destOverride: ["fakedns", "tls"]`, so Xray reads the TLS SNI from each connection to determine the real destination before routing.

**`tls-decrypt-h11` (port `11666`)** — A `dokodemo-door` inbound that acts as a transparent TLS endpoint. It receives redirected TCP/443 traffic and terminates TLS using your `mycert.crt` and `mycert.key`. The `usage: "issue"` certificate setting means Xray dynamically generates a per-domain certificate signed by your CA, so your browser sees a trusted certificate for each site. Only `http/1.1` is offered via ALPN.

**`tls-decrypt-h211` (port `11777`)** — Identical to `tls-decrypt-h11` but negotiates both `h2` and `http/1.1` via ALPN. This inbound handles main Google services (Meet, Drive, etc.) that benefit from HTTP/2 multiplexing.

<Note>
Both MITM inbounds use `followRedirect: true` so they correctly inherit the original destination address from the routing redirect.
</Note>

## Outbounds

Seven outbounds cover every traffic path: blocking, direct connections, DNS, local redirects to the MITM inbounds, and TLS re-encryption for Domain Fronting.

| Tag | Protocol | Purpose |
|---|---|---|
| `block` | `block` | Drop the connection entirely |
| `direct` | `direct` | Send traffic directly to the internet |
| `dns-out` | `dns` | Handle DNS queries via Xray's DNS engine |
| `redirect-out-h11` | `direct` | Redirect TCP/443 to the `tls-decrypt-h11` MITM inbound |
| `redirect-out-h211` | `direct` | Redirect TCP/443 to the `tls-decrypt-h211` MITM inbound |
| `tls-repack-google-h11` | `direct` | Re-encrypt with spoofed SNI, HTTP/1.1 (googlevideo.com) |
| `tls-repack-google-h211` | `direct` | Re-encrypt with spoofed SNI, HTTP/2 (Google main services) |

**`block`** — Silently drops any matched connection. Used for ad domains.

**`direct`** — Sends traffic directly to the resolved IP using `domainStrategy: "ForceIP"` (always resolves the domain to an IP before connecting). Happy Eyeballs is enabled to race IPv4 and IPv6 connections simultaneously for the fastest path.

**`dns-out`** — Handles DNS lookups through Xray's DNS subsystem. Non-IP queries (e.g. `MX`, `PTR`) are rejected, and record types `0` and `65` (QTYPE=ANY and HTTPS/SVCB) are blocked to prevent DNS-based fingerprinting.

**`redirect-out-h11`** — A `direct` outbound that redirects the connection to `127.0.0.1:11666` instead of the real destination. This feeds traffic into the `tls-decrypt-h11` MITM inbound for local TLS decryption.

**`redirect-out-h211`** — Same as above but redirects to `127.0.0.1:11777` for the HTTP/2 MITM pipeline.

**`tls-repack-google-h11`** — After decryption, this outbound re-encrypts the traffic for the upstream server. It sets `serverName: "www.google.com"` as the TLS SNI (Domain Fronting — censors see an allowed domain), uses Chrome's TLS fingerprint, and restricts ALPN to `http/1.1`. Used for `googlevideo.com` video streaming traffic.

**`tls-repack-google-h211`** — Same Domain Fronting technique but supports `h2` and `http/1.1` ALPN. Targets Google main services by redirecting to `google-main-services.redirect:443`, which the DNS section resolves to `216.239.38.120` (a Google anycast IP).

## DNS configuration

```json
"dns": {
  "hosts": {
    "geosite:category-ads-all": "#3",
    "google-main-services.redirect": "216.239.38.120"
  },
  "servers": ["fakedns", "localhost"],
  "queryStrategy": "UseSystem",
  "useSystemHosts": true,
  "tag": "dns_inbound"
}
```

Xray resolves DNS in order: `fakedns` first (for transparent proxy sniffing), then `localhost` (your system resolver) as a fallback.

Two static `hosts` entries override normal resolution:

- `geosite:category-ads-all → "#3"` — Maps all ad domains to an invalid address, effectively blocking them before any connection is attempted.
- `google-main-services.redirect → 216.239.38.120` — A synthetic hostname used internally by the `tls-repack-google-h211` outbound. It resolves to a Google anycast IP, ensuring HTTP/2 Google traffic reaches a real Google server.

The DNS module is tagged `dns_inbound` so routing rules can target it directly (DNS queries from the DNS engine itself go `direct`, not through `dns-out`).

## Logging

```json
"log": {
  "loglevel": "warning",
  "dnsLog": false,
  "access": "none"
}
```

- **`loglevel: "warning"`** — Only warnings and errors are logged. Set to `"debug"` temporarily if you need to troubleshoot routing decisions.
- **`access: "none"`** — Access logs are disabled to avoid writing a log file for every connection.
- **`dnsLog: false`** — DNS query logging is off. Enable it during debugging to inspect which domains are being resolved and by which server.
